# Atlas Rules

## When to use atlas/

This is the zone for **maps, indexes, and referenceable entities**. It is the workspace's meta layer, the one that lets you navigate by idea rather than by folder.

## Structure

- `atlas/people/{slug}.md`, people files (contacts, decision-makers, prospects, individual partners)
- `atlas/services/{slug}.md`, your offerings (web-audit, managed-hosting, strategy-consulting, etc.)
- `atlas/topics/{slug}.md`, cross-cutting subjects (ai-agents, kubernetes, seo, accessibility, supply-chain-security)
- `atlas/maps/{slug}.md`, thematic MOCs OR Dataview indexes, distinguished by `subtype: moc | index`
- `atlas/home.md`, main hub, the workspace's single entry point

## Types and schemas

| File | BM type | BM schema | Validation |
|---|---|---|---|
| `atlas/people/*.md` | `person` | `basic-memory/schemas/person.md` | strict |
| `atlas/services/*.md` | `service` | `basic-memory/schemas/service.md` | strict |
| `atlas/topics/*.md` | `topic` | `basic-memory/schemas/topic.md` | strict |
| `atlas/maps/*.md` | `note` + `subtype: moc \| index` | `basic-memory/schemas/note.md` | warn |
| `atlas/home.md` | `note` + `subtype: moc` | same | warn |

## MOC (Maps of Content)

Manual thematic maps for dense subjects that deserve an entry page. For the template, see `templates/atlas-moc-template.md`.

Naming convention, `MOC-{subject}.md` or `{subject}-moc.md`:

- `atlas/maps/MOC-ai-agents.md`
- `atlas/maps/MOC-kubernetes.md`
- `atlas/maps/MOC-workspace-methodology.md`

## Dataview indexes

Lists generated dynamically by Dataview in Obsidian, plus `*-static.md` versions generated by script for the agents.

Automatically generated indexes:

- `atlas/maps/clients-active.md` + `clients-active-static.md`
- `atlas/maps/partners-active.md` + `partners-active-static.md`
- `atlas/maps/projects-active.md` + `projects-active-static.md`
- `atlas/maps/recent-emails.md` + `recent-emails-static.md`
- `atlas/maps/decisions-recent.md` + `decisions-recent-static.md`

Regeneration script, `python3 basic-memory/scripts/generate_indexes.py`. Run it again after a significant writing session or weekly.

**Portability**, Dataview queries only run inside Obsidian. The `*-static.md` versions are readable outside Obsidian (any editor, agents). If you leave Obsidian, regenerate the statics and delete the Dataview versions.

## MOC frontmatter

```yaml
---
title: MOC, {Subject}
type: note
schema_version: 1
subtype: moc
date: YYYY-MM-DD
status: active
topics: [topic1, topic2]
---
```

## Index frontmatter

```yaml
---
title: {Index name}
type: note
schema_version: 1
subtype: index
date: YYYY-MM-DD
status: active
auto_generated: true   # for -static.md versions only
---
```

## Wiki-link conventions

MOCs contain **dense** wiki-links to:

- Modeled entities (`[[Person]]`, `[[Service]]`, `[[Topic]]`)
- Clients or partners involved (`[[Globex]]`, `[[Initech]]`)
- Active projects (`[[Globex Website Audit]]`)
- Distilled knowledge (`[[playbook-client-onboarding]]`)
- External sources (URLs)

The goal is for a MOC to act as a visual aggregation node in the Obsidian graph.

## Obsidian color groups

To make the graph more readable, configure color groups:

- `type:client-context` -> green
- `type:partner-context` -> dark green
- `type:meeting` -> blue
- `type:email` -> gray
- `type:person|service|topic` -> orange
- `type:note` + `subtype:pattern|playbook|guide` -> red
- `type:project-brief` -> purple
- `type:note` + `subtype:moc|index` -> black/outline

## Maintenance

- MOCs are not created by default; they emerge when a subject becomes recurring and deserves an entry point.
- MOCs decay over time; run an annual review to archive the ones no longer in use.
- Dataview indexes regenerate automatically when opened in Obsidian.

## Forbidden rules

- No raw artifact (email, transcript, meeting notes) in `atlas/`
- No wikilink in the `name:` or `title:` frontmatter (unless the value is deliberately text)
- No MOC without relations in its body (a MOC with no links aggregates nothing)
