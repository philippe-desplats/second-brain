# Save Conventions

Universal save logic for all skills that produce deliverables. Every skill that generates a file MUST follow these rules.

## Absolute Rule

**NEVER save a file at the root of a client folder.** Every file lives inside a project:

- `sources/clients/{client}/{project}/...` (correct)
- `sources/clients/{client}/file.md` (forbidden)
- `sources/clients/{client}/emails/` (forbidden)

## Path Resolution

```
1. DETECT the client
   - From prompt, recipient, context, or loaded CLAUDE.md (fallback: CONTEXT.md)
   - If not found, ASK the user

2. RESOLVE the project
   - List existing projects: ls sources/clients/{client}/*/
   - 1 project  -> use it automatically
   - N projects -> infer from context (topic, recipient, subject), ask if ambiguous
   - 0 projects -> ask the user for a project name, create the folder

3. DETERMINE the subfolder
   - See table below

4. VERIFY the folder exists
   - If not, create it (mkdir -p)

5. NAME the file
   - ALWAYS: YYYY-MM-DD-{type}-slug.md
   - Date: today's date
   - Type: from the table below
   - Slug: kebab-case, descriptive, short

6. SAVE and confirm the path to the user
```

## Subfolder and Type Mapping

| Content | Subfolder | `type:` frontmatter | Filename prefix |
|---------|-----------|----|-----|
| Email (new, reply) | `emails/` | `email` | see Email Convention below |
| Meeting notes / minutes | `meetings/` | `meeting` | `meeting` |
| Raw transcript | `transcripts/` | `transcript` | `transcript` |
| Contract / quote / signed deliverable | `deliverables/` | `deliverable` | `writing` |
| Research / audit / benchmark / watch | project root | `research` | `research` |
| Brainstorm / ideation | project root | `brainstorm` | `brainstorm` |
| Strategy / roadmap / plan | project root | `strategy` | `strategy` |
| Writing in progress (draft articles, posts) | project root | `writing` | `writing` |
| Note / synthesis / aide-memoire | project root | `note` | `note` |
| Project brief / README | project root | `project-brief` | `README` |

**CRITICAL**: the `type:` field in the YAML frontmatter is **mandatory** for Basic Memory indexing. Without it, the file is not recognized as belonging to a canonical type and cannot be validated against a schema. Canonical schemas live in `basic-memory/schemas/{type}.md`.

## Email Convention

Emails follow a specific naming convention for ordering and traceability:

```
YYYY-MM-DD-{seq}-{direction}-{slug}.md
```

| Field | Description | Values |
|-------|-------------|--------|
| `seq` | Sequential number for the day | `01`, `02`, `03`... |
| `direction` | Email direction | `out` (sending), `in` (received), `reply` (response) |
| `slug` | Short kebab-case description | e.g., `commercial-proposal` |

### Email frontmatter (required)

```yaml
---
type: email
date: YYYY-MM-DD
time: "HH:MM"
seq: 1
direction: out | in | reply
status: draft | sent | cancelled
from: Full Name
to:
  - Full Name
  - Full Name 2
cc:
  - Full Name
subject: Email subject line
thread: thread-identifier (optional)
---
```

**`type: email`** is mandatory; without it the file is not indexed as an email. `permalink:` is auto-generated by `bm reindex`, no need to enter it. `to` and `cc` must be YAML arrays (one recipient per line with `-`), not strings separated by `;`.

### Status values

- `draft`: being written, not yet sent
- `sent`: sent to recipient
- `cancelled`: abandoned (file kept for traceability)

### Examples

```
emails/2026-04-06-01-out-commercial-proposal.md
emails/2026-04-06-02-reply-quote-validation.md
emails/2026-04-06-03-out-content-follow-up.md
```

### Migration note

Older emails may follow the legacy convention `YYYY-MM-DD-{writing|reply}-slug.md` without sequence numbers or structured frontmatter. Do not rename existing files. Apply the new convention to all new emails going forward.

## Base frontmatter for the other types

Every non-email type follows a minimal schema (see `basic-memory/schemas/{type}.md` for the full fields):

```yaml
---
title: Descriptive title of the content
type: meeting | research | strategy | brainstorm | writing | note | deliverable | transcript | project-brief
schema_version: 1
date: YYYY-MM-DD (for dated content)
status: active (optional)
---
```

For content tied to an entity (client meeting, client research, etc.), add:

```yaml
client: Canonical client name (if client scope)
partner: Canonical partner name (if partner scope)
source: Recording URL or external source (for meeting, transcript, research)
```

And in the body, a `## Relations` section with `- about [[Canonical name]]` pointing to the parent client or partner. This densifies the Obsidian graph and enables Basic Memory navigation.

## For internal projects

Same logic but the path is `sources/internal/{project}/...`. For files not tied to a client or partner, add `- about [[Acme]]` in the Relations section, where `Acme` is the placeholder for your own organization (rename it to your company, described in `sources/internal/CONTEXT.md`).

## For personal projects

Same logic but the path is `sources/personal/{project}/...`. Relations optional.

## Examples

```
/ops-email for Globex about site progress
-> sources/clients/globex/website-redesign/emails/2026-03-13-01-out-progress-update.md (type: email)

/ops-meeting-notes of a meeting transcript with Globex
-> sources/clients/globex/website-redesign/meetings/2026-03-13-meeting-weekly-sync.md (type: meeting)

/ops-seo-audit for Initech intranet
-> sources/clients/initech/legacy-migration/2026-03-13-research-seo-audit.md (type: research)

/ops-web-benchmark -p globex
-> sources/clients/globex/website-redesign/2026-03-13-research-web-benchmark.md (type: research)

/ops-sale-quote for new prospect
-> sources/clients/{prospect}/{project}/deliverables/2026-03-13-writing-quote.md (type: deliverable)

/ops-proposal for Initech
-> sources/clients/initech/legacy-migration/2026-03-13-writing-proposal.md (type: writing)

/ops-contract for Globex
-> sources/clients/globex/contracting/deliverables/2026-03-13-writing-contract.md (type: deliverable)
```
